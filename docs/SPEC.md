# code-context MCP 服务 — 产品背景与需求规格

## 1. 产品背景

code-context MCP 服务是为 AI 编码助手提供代码上下文的 MCP (Model Context Protocol) 服务，通过语义搜索、符号查找、影响分析等能力，使 AI 在辅助需求变更、调试 bug 过程中能真正减少上下文调用，替代 Grep/Glob/Read 的组合使用。

### 1.1 核心问题

| 问题 | 影响 |
|------|------|
| `index_project` 需手动调用，AI 不会主动触发 | 搜索经常返回空结果，MCP 形同虚设 |
| `SplitText` 按 800 字符硬切，不考虑代码结构 | 函数被切成两半，语义断裂，搜索质量差 |
| 仅支持语义搜索，精确符号查找不如 Grep | 查找字段引用仍需 Grep |
| `file_context` 返回完整文件内容 | 大文件消耗大量上下文窗口 |
| 缺少影响范围分析能力 | 重构时需大量 Grep + Read 组合 |

### 1.2 功能范围

**包含：**
- 自动索引机制（启动时检测 + 搜索时过期更新）
- 按代码结构切分（Go/Vue/JS/TS/MD/Py）
- symbol_search 精确符号搜索工具
- impact_analysis 影响范围分析工具
- file_context 结构化摘要模式

**不包含：**
- 完整 AST 解析器引入（使用正则匹配即可满足大部分场景）
- 跨项目索引支持
- 向量数据库更换（继续使用 Zilliz Cloud）
- MCP 协议版本升级
- 嵌入模型更换

## 2. 需求规格

### 2.1 P0：自动索引

MCP 服务启动时自动检测索引状态，搜索时发现索引过期自动触发后台增量更新，使 AI 无需显式调用 `index_project` 即可使用 `code_search`。

| ID | 需求描述 | 优先级 |
|----|---------|--------|
| REQ-001 | 启动时自动检测索引状态 | P0 |
| REQ-002 | 索引不存在时自动触发全量索引构建 | P0 |
| REQ-003 | 索引存在但检测到代码变更时标记索引为过期 | P0 |
| REQ-004 | 搜索请求发现索引过期时后台自动触发增量更新 | P0 |
| REQ-005 | 索引正在后台更新时仍可使用现有索引响应搜索请求 | P0 |
| REQ-006 | 记录上次索引的 git commit hash 或文件 mtime 摘要作为索引状态标识 | P0 |
| REQ-007 | 增量更新完成后更新索引状态标识 | P0 |

### 2.2 P1：按代码结构切分

替换固定字符窗口切分为按代码语法结构切分，使每个切分块对应完整的语法单元（函数、结构体、组件等），保留上下文元数据。

| ID | 需求描述 | 优先级 |
|----|---------|--------|
| REQ-008 | 按代码语法结构边界切分代码，而非固定字符窗口 | P1 |
| REQ-009 | Go 代码按 func、type struct、type interface、var/const 块边界切分 | P1 |
| REQ-010 | Vue 代码按 template/script/style 块切分，script 内按函数/组件切分 | P1 |
| REQ-011 | JS/TS 代码按 function、class、export、const 块边界切分 | P1 |
| REQ-012 | Markdown 按标题（#/##/###）切分 | P1 |
| REQ-013 | 每个切分块保留上下文元数据（file、symbol、line_start、line_end、type） | P1 |
| REQ-014 | 超长块进行二次切分 | P1 |

### 2.3 P1：symbol_search 精确符号搜索

增加基于倒排索引的精确符号搜索工具，支持按符号名查找定义和引用，返回结构化结果。

| ID | 需求描述 | 优先级 |
|----|---------|--------|
| REQ-015 | 提供 symbol_search 工具 | P1 |
| REQ-016 | 返回匹配符号名的所有出现位置 | P1 |
| REQ-017 | search_type=definition 仅返回定义位置 | P1 |
| REQ-018 | search_type=reference 仅返回引用位置 | P1 |
| REQ-019 | search_type=all 返回定义和引用位置 | P1 |
| REQ-020 | 结果按文件分组，包含文件路径、行号和上下文摘要 | P1 |
| REQ-021 | 支持驼峰/下划线风格转换的模糊匹配 | P1 |
| REQ-022 | 索引构建时同步构建符号倒排索引 | P1 |

### 2.4 P2：impact_analysis 影响范围分析

增加影响范围分析工具，一次调用完成符号删除/重命名/修改签名的影响范围分析。

| ID | 需求描述 | 优先级 |
|----|---------|--------|
| REQ-023 | 提供 impact_analysis 工具 | P2 |
| REQ-024 | action=delete 返回受影响位置及删除建议 | P2 |
| REQ-025 | action=rename 返回需替换位置及替换建议 | P2 |
| REQ-026 | action=modify 返回受影响的调用位置 | P2 |
| REQ-027 | 每个影响位置包含文件、行号、影响类型、上下文和修改建议 | P2 |
| REQ-028 | 返回影响汇总（受影响文件总数、引用总数、分类统计） | P2 |
| REQ-029 | 复用 symbol_search 的倒排索引 | P2 |

### 2.5 P2：file_context 结构化摘要

为 file_context 工具增加 mode 参数，支持 summary 模式返回文件结构摘要。

| ID | 需求描述 | 优先级 |
|----|---------|--------|
| REQ-030 | file_context 支持 mode 参数 | P2 |
| REQ-031 | mode=full 返回完整文件内容 | P2 |
| REQ-032 | mode=summary 返回文件结构摘要 | P2 |
| REQ-033 | summary 返回函数列表（含名称和行号范围） | P2 |
| REQ-034 | summary 返回类型列表（含名称、类型和行号） | P2 |
| REQ-035 | summary 返回导入依赖列表 | P2 |
| REQ-036 | summary 返回总行数和语言类型 | P2 |

## 3. 非功能需求

| ID | 需求描述 | 指标 |
|----|---------|------|
| NFR-001 | 索引状态检测耗时 | 启动时不超过 5 秒 |
| NFR-002 | 增量更新耗时 | 不超过全量重建的 30% |
| NFR-003 | symbol_search 响应时间 | 不超过 100ms |
| NFR-004 | impact_analysis 响应时间 | 不超过 2 秒 |
| NFR-005 | summary 模式响应时间 | 不超过 500ms |
| NFR-006 | 后台索引更新失败不影响搜索服务 | 搜索请求始终可响应 |
| NFR-007 | 索引状态持久化 | 异常退出后重启不丢失 |
| NFR-008 | 倒排索引一致性 | 与向量索引同步构建 |
| NFR-009 | 向后兼容 | 现有工具行为不变 |
| NFR-010 | MCP 协议兼容 | 新增工具遵循现有规范 |
